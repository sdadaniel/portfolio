## Summary

금융 서비스 환경에서 **하이브리드 앱(App·WebView)** 구조를 안정화하고, **인증·딥링크·실시간 기능**과 **운영형 UI**를 프로젝트 단위로 개선해 사용자 경험과 운영 효율을 함께 끌어올린 경험.

## 1. React Native 기반 하이브리드 앱 운영 — 인증·딥링크·WebView–Native 연동

### 개요

- **React Native** 기반으로 앱을 운영했으며, **생체 인식**, **법인 로그인**, **메인 화면** 등 앱 화면·플로우 제작
- 사용 중이던 **라온**, **코스콤** 모듈을 **신규 모듈로 교체**
- 교체 과정에서 **Swift / Objective-C / Kotlin** 기반 네이티브 코드 수정·연동 처리

### 결과

- 인증 모듈 교체를 통해 **보안 요구사항을 충족하면서 안정적인 인증 환경** 구축
- **생체 인식·법인 로그인·메인** 등 핵심 화면을 **네이티브–웹 흐름과 맞물리게** 정리해, **진입·인증 UX**를 한 줄로 이어지게 함

## 2. MFE(Micro Front End)

### Problem

- WebView 쪽은 원래 **Vue3**로 작성된 화면이 중심이었고, **마이크로 프론트엔드(MFE)** 를 전제로 **React 쪽으로 전환** 이 필요함
- **렌더링·런타임 등 성능**을 기준으로 볼 때 **Vue보다 React가 유리하다**고 판단해, 점진적으로 React 스택으로 갈아타는 방향을 잡음
- 전환 구간에는 **Vue와 React가 한 앱·한 화면에서 공존**해야 하고, 기존 **Vuex**·운영 흐름과 **자연스럽게 연동**돼야 함
- 화면·기능이 늘면서 **번들·로딩** 부담과 수정 범위도 함께 커져, 전환과 **번들 최적화**를 같이 고려해야 함

### Approach

- **React 컴포넌트를 Web Component로 감싸** `react-dom`과 `customElements.define()`으로 마운트해, **Vue 템플릿에서 `<custom-react-button>` 형태로 직접 사용**할 수 있게 구성
- **속성(attribute) ↔ props**를 맞춰 **양방향 데이터 흐름**을 설계하고, Vue 쪽으로는 **`CustomEvent` 디스패치**로 클릭·변경 등 이벤트를 넘겨 **Vue↔React 경계를 명확히** 정리
- **점진적 리팩터링** 전략: 먼저 **Web Component로 이식**해 화면별 치환 리스크를 줄인 뒤, 필요 시 **React 컴포넌트 마이그레이션**으로 이어질 수 있는 구조
- **스타일 캡슐화**(Shadow DOM·스코프 전략 등)로 **공통 CSS와의 충돌**을 막고, 레거시 Vue 스타일과 새 UI가 겹치지 않도록 정리

### Result

- React UI를 **Vue 앱 안에서 안전하게 끼워 넣을 수 있는** 기술적 토대 마련
- **대규모 일괄 전환 없이** 화면별 **단계적 전환**이 가능한 구조로 정리


### 회고

- **MFE**로 가더라도 **host(Vue) 쪽 뼈대를 크게 바꾸기 어렵다**는 제약이 있었고, 그 안에서 React 조각을 얹는 형태에 가까웠음
- **전역 상태**를 Web Component 경계 너머로 맞추는 데 공수가 들었고, **Shadow DOM**으로 **스타일 충돌**은 줄일 수 있었으나 **공통 디자인 시스템**을 그대로 끌어다 적용하기는 까다로웠음
- 예상보다 **보일러플레이트 코드**가 많아짐
- 빠른 이전을 위해 **Web Component**를 택했지만, 비슷한 과제를 다시 한다면 **Module Federation**으로 가져가면 **경계·운영 측면에서 더 나은 경험**을 만들 수 있을 것 같음

## 3. WebSocket 기반 실시간 채팅·거래 기능 개발

### 개요

- **WebSocket**으로 **실시간 채팅·거래(협의·체결)** 구간의 이벤트를 push 기반으로 처리하고, 화면·세션 상태를 **서버 이벤트와 동기화**
- **장 내·장 외** 등 운영 규칙 차이에 맞춰 **동일 프로토콜 위에서 분기**할 수 있게 조건·검증 플로우를 구성

### 결과

- 실시간 UX 요구를 충족하는 기능 라인을 안정적으로 유지·발전시킴

## 4. 운영용 에디터(Tiptap)·설정 및 도메인 화면 개선

### Problem

- 당시 쓰던 운영용 에디터는 **WYSIWYG가 아니**라 **작성 화면과 실제 서비스 노출이 어긋나기 쉬웠고**, 맞는지 보려면 **실제 운영 환경에서 다시 확인**해야 하는 **불편**이 있었음
- 운영 콘텐츠·설정 흐름의 UX 제약과, 도메인 특화 화면(예: **수익자총회**)의 판매·결과 반영 로직 정리도 필요함

### Approach

- **Tiptap**(내부 **ProseMirror** 기반)으로 에디터를 구축·커스터마이징해 **편집 중 보이는 결과와 운영(서비스) 화면 노출을 일치**시키는 방향으로 맞춤
- 테이블 균등 너비, 이미지 100% 폭, 캡션, 플레이스홀더, 문단 전환 등 **콘텐츠 블록**을 다루고, 링크 버튼·텍스트 배경색 등 **운영 UX**를 보강
- 설정 핵심 화면 **전면 리뉴얼**(IA·UX 재정비), 수익자총회 **페이지 구조 개선** 및 판매·결과 반영 로직 구현

### Result

- **편집 단계에서 레이아웃·스타일을 거의 그대로 검증**할 수 있어, **운영 배포 후 재확인에 쓰이던 공수**를 줄임
- 에디터·설정·도메인 화면의 반복 이슈를 줄이고 운영 효율을 높이는 방향으로 개선함
