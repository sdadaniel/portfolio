# 플라잉캣

**2021.07 - 2023.02** · 정규직 · 퀵커머스(10분특공대)·스마트 스케줄링(Sync) 등 서비스별 풀스택·프론트 주도

## Summary

**퀵커머스(10분특공대)** 는 **모바일 앱**과 **운영 관리자**를 프로젝트 단위로 나눠 앱·백오피스·API까지 **소수 인력 E2E**로 맞추고, 이후 **일정·캘린더(Sync)** 서비스에서 기획부터 다국어·성능·테스트·배포까지 세팅한 경험.

**10분특공대** 기간에는 개발만 한 것이 아니라 **실제 배달**을 직접 하며 현장에서 서비스 **문제점을 몸으로 겪고**, 그때 느낀 불편·이슈를 바탕으로 앱·관리자·운영 흐름을 **직접 고쳐 나감**. 예를 들어 매장 안에서 **어떤 물품이 어디에 있는지** 찾기 어렵다는 걸 겪고, **주문이 들어오면** 해당 품목이 **어느 선반**에 있는지 바로 보이도록 **표시하는 기능**을 추가하는 식으로 반영함.

## 1. 퀵커머스 「10분특공대」모바일 앱 (2022.05 - 2022.10)


`#ReactNative` `#ReactQuery` `#TypeScript` `#Payple` `#Express` `#MySQL`

### 개요
모바일 앱의 **초기 설계부터 개발, 배포·운영까지** 경험하며 리뷰·상품·장바구니 등 핵심 커머스 흐름을 구현하고, 결제·정산은 외부 PG 연동으로 연결함. API 통신·캐싱 전략으로 카테고리 등 목록 로딩 체감 속도를 개선하고, 같은 서비스의 백엔드와 스키마·응답 규약을 맞춰 앱–서버 데이터 구조를 일원화. **직접 배달**을 하며 겪은 현장 이슈를 근거로 화면·흐름을 손봄. 매장에서 **물품 위치**를 찾기 어렵다는 문제를 겪고, **주문이 들어오면** 품목마다 **어느 선반**에 있는지 **표시**하는 기능을 추가해 피킹·출고에 쓰이게 함.

### 결과

- 카테고리 **캐싱**으로 체감 속도 개선, 앱–서버 **데이터 구조 일원화**로 오류·유지보수 부담 감소
- **배달 현장**에서 드러난 문제를 먼저 체감한 뒤, 그에 맞춰 기능·UX를 수정·보완
- 매장 내 **물품 탐색**이 어렵다는 점을 반영해, **주문 단위로 품목별 선반 위치**를 보여 주는 흐름을 넣어 찾는 시간을 줄임


### 회고

- **Node.js 백엔드** 업무를 맡고 있던 중 **모바일 프론트 담당 공백(퇴사 등)**이 생기며 **백엔드에서 React Native 앱 개발**로 전환해 프로젝트에 합류함.
- **웹 React 경험 없이** React Native를 시작해 **초기 환경 세팅부터** 진입 장벽이 컸지만, 당시 팀이 필요로 하는 범위에 **실질적으로 기여**할 수 있어 **뿌듯함**으로 남음.
- **첫 앱 프로젝트**라 부족함이 많았지만 동료 덕분에 빠르게 익혔고, **몇 달 안에 단독으로 운영·배포**까지 맡을 수 있을 만큼 성장한 **의미 있는 기간**이었음.

## 2. 10분특공대 관리자 페이지 제작  (2022.05 - 2022.10)


`#Node` `#Express` `#MySQL` `#Vue` `#BootstrapVue` `#KakaoMap` `#Polygon`

### 개요

앱·관리자가 함께 쓰는 API 서버를 구축하고, 운영용 관리자 화면에서 상품·주문·고객 등 핵심 운영 업무를 풀스택으로 설계·구현. **MFC별**로 지역을 나눠 **매장**을 관리할 수 있도록 하고, **카카오 지도**를 붙여 지역·매장 위치를 지도 위에 표시·관리하는 기능을 넣음. 매장 **영역**은 **polygon**을 기준으로 지도에 그려 **영역 표시**·관리까지 할 수 있게 함. **실제 배달**을 병행하며 운영·지도·영역에서 생기는 **문제를 현장에서 확인**하고, 관리자 기능과 로직을 그에 맞게 **수정·보완**함. 상품마다 **선반·위치** 정보를 등록·관리해 두고 주문·앱 화면과 맞물리게 하여, **주문 시 품목별 선반 표시**가 돌아가도록 연동함.

### 결과

- 앱과 동일 API·데이터 모델 위에서 **운영–앱 End-to-End** 흐름 유지, 소수 인력으로 개발·운영 가능한 구조 확보
- **MFC·지역 단위 매장**을 **카카오 지도**로 시각화하고, **polygon** 기준 **매장 영역**도 표시·관리해 운영자가 위치·담당 구역을 파악할 수 있게 함
- 배달 현장 경험과 관리자 도구를 맞물려, **체감한 문제**가 반영된 운영 기능으로 다듬음
- 상품–**선반(위치)** 데이터를 관리자에서 다루고 주문과 연결해, 매장에서 물건을 찾기 어렵다는 현장 이슈를 줄이는 데 기여

### 회고

- **첫 Node.js 풀스택 프로젝트**라 러닝커브가 있었지만, 관리자 화면은 **Vue**로 구성해 요구사항을 빠르게 구현·반영할 수 있었음.
- **Kakao Map 등 외부 라이브러리**를 적용하면서 **OpenAPI 규격**과 **RESTful 문서**를 꼼꼼히 참고해, 사내 서비스 구조·규칙에 맞게 정리·적용하려고 노력한 경험이었음.

## 3. Sync 스마트 스케줄링 (2022.10 - 2023.02)

### Stack

Next.js, TypeScript, MUI, Recoil, Storybook, Jest, FullCalendar, next-i18next, ESLint, Husky, Jenkins, GitHub Actions, AWS ECR, React Testing Library

### 키워드

`#Nextjs` `#TypeScript` `#MUI` `#Recoil` `#Storybook` `#Jest` `#FullCalendar` `#i18n` `#ESLint` `#Husky` `#Jenkins` `#GitHubActions` `#AWSECR` `#RTL` `#Template` `#DesignSystem`

### 개요

**신규 프로젝트**로 스마트 스케줄링 서비스 **Sync**를 진행. **빠른 신규 구축**과 **안정성**을 동시에 맞추기 위해 **프로젝트 템플릿**을 쓰고, 그에 맞게 **디자인 시스템**을 수정·정비함(일정·캘린더 도메인에 맞춤). 기획·초기 설계부터 일정·캘린더 UI/UX, 다국어, 성능, 테스트, 배포·자동화 파이프라인까지 도메인에 맞춰 전 과정을 담당.

### 결과

- 재사용 가능한 UI·문서화 체계와 **성능·배포·테스트**가 갖춘 스케줄링 제품 기반 마련
