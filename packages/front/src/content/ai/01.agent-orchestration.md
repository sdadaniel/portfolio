## 1. 개요

agent 지능이 높아지면서 개발 생산성이 전과는 비교할 수 없을정도로 향상되었다. 하지만 다중 agent들을 효율적으로 다루는 일은 쉽지 않다. 여러 터미널에 클로드 화면을 띄워놓고 개발하는 모습이 익숙하지만 개발 멀티 태스킹으로 작업하는 부분이 개발자의 피로도도 높이고 있다. 그래서 웹단에서 클로드를 orchestrate하는 서비스를 개인적으로 만들어보았다. 이를 통해 24시간 업무 진행, 할당, 개선을 스스로 할 수 있는 서비스를 만들고자 하였다.

## 2. 문제


- **인간 병목**: 에이전트보다 **프로세스 전체에서 가장 느린 축이 사람** — 확인·승인·질문·지시가 끊기지 않아야만 다음으로 넘어가서, **전체 처리량이 인간의 개입 속도·가용 시간에 묶임**
- **상시 운영 한계**: 에이전트를 24시간 돌리기 어렵고, 코드는 생성해 줘도 **테스트·검증은 사람**에게 맡겨지는 경우가 많음
- **컨텍스트·비용**: 사람이 계속 대화를 끌고 가다 보면 **맥락이 길어지고**, 그만큼 **토큰·비용**도 불필요하게 커짐
- **병렬 작업의 충돌**: 여러 agent가 같은 파일을 동시에 수정하면 충돌이 빈번하고, 수동 해결에 시간이 소모됨

## 3. 해결방법

### 3-1. 인간 병목
  - **Task Queue 활용**
    - 핵심은 사람이 **에이전트 작업이 끝날 때까지 기다리지 않고** **태스크를 계속 큐에 넣을 수 있다**는 점임. **Task Queue**에 할 일을 미리 쌓아 두면, 에이전트는 순서대로 처리하고 사람은 **완료 대기·중간 확인**에 묶이지 않음.
    <img src="/ai/agent-orchestration/taskqueue.png" alt="Agent 오케스트레이션 멀티 패널 화면" style="width:500px;max-width:100%;display:block;margin:0 auto" />

  - **PR 없이 바로 머지**
    - 여기서 **즉시 머지**는 **dev/main에 곧장 넣는다**는 뜻이 아님. **에이전트 전용 브랜치**를 두고, 작업이 끝나면 **AI review**만 거쳐 **그 브랜치에 바로 머지**해 PR·사람 승인 대기에 묶이지 않게 함. **dev·main**(또는 배포 기준 브랜치) 반영은 이후 **사람의 검수**를 거쳐 머지하는 전략을 취함. 
    - 전제는 **에이전트가 사람보다 코드를 더 잘 쓰고 리뷰도 잘한다**는 점 — 에이전트도 오류는 낼 수 있지만 **에이전트가 바로잡을 수 있게** 두고(review 실패 시 **자동 retry**, 한도 초과 시 **failed**). 사람은 **방향·우선순위**와 **최종 병합**만 잡고 일상적 확인·지시는 최소화.
    


### 3-2. 상시 운영 한계
  - 사람은 하루 **8시간** 정도만 일할 수 있어도, 에이전트는 **24시간** 돌릴 수 있음 — 이 차이를 살리려면 **Task Queue**에 **충분한 태스크를 넣어 두는 것**이 핵심. 큐에 일이 쌓여 있으면 사람이 자리를 비워도 에이전트가 이어서 처리해 **24시간 가동**이 가능해짐. 
  - **태스크 우선순위**를 조절해 **사람 검수가 필요한 태스크**는 사람이 있는 시간에 돌리고, **상대적으로 단순한 태스크**는 그 외 시간(야간·자리를 비운 시간 등)에 돌림. 후자에는 **테스트 코드 보강**, **ESLint·코드 품질 점검**, **커버리지** 같은 항목이 포함됨.


### 3-3. 컨텍스트·비용
  - **태스크 단위**로 나누고, 태스크마다 **에이전트 세션을 초기화**해 한 채팅 스레드에 맥락을 쌓지 않으며 이전 작업이 다음으로 **끌려오지 않게** 함. **AI 리뷰**도 구현과 **별도 세션**으로 돌려, 작업 중 쌓인 대화 맥락이 **리뷰 단계로 끌려가지 않게** 함. 정의 시 **scope**로 넘길 컨텍스트를 좁히고, **마크다운**에 상태·로그를 남겨 **토큰·대화 비용**을 줄임.
  - **태스크를 지속적으로 모니터링**하며 로그·비용으로 토큰·재시도·중복 작업 등 **새는 지점**을 보고 **절감안을 적용·고안**함.

### 3-4. 병렬 작업의 충돌
  - 병렬로 돌리려면 **충돌이 나지 않는 것**이 가장 중요함. 태스크를 정의할 때 **scope**를 명시하고, **scope이 겹치지 않는 조합**만 **동시에** 돌림. 
  - **git worktree**로 태스크마다 격리된 작업 트리를 쓰고, **scope 충돌 검사**로 같은 파일을 건드리는 태스크는 **순차 실행**해 머지 충돌·수동 해결 부담을 줄임.

## 5. 개발
### 5-1. Task 등록 프로세스

```mermaid
flowchart LR
    A[1.Task 작성] --> B{유효한 Task인가?}

    B -->|아니오| C[Task 거절]
    B -->|예| T[2.Task 생성]
    T --> D{3.큰 작업인가?}

    D -->|예| E[다중 Task]
    D -->|아니오| F[단일 Task]

    E --> G{4.Task 승인}
    F --> G

    G -->|아니오| H[Task 거절]
    G -->|예| I[5.Task 등록 - Pending]
```
#### 1. Task 작성

대시보드에서 태스크를 만들고 제목·설명 등을 짧게 적는다. 에이전트에 넘기기 **전에** 요구사항을 구체적으로 정리해 둔다.

#### 2. Task 생성

유효성 검사를 통과하면 **태스크 정의**가 생성된다. 예를 들어 `TASK-*.md` 형식과 YAML frontmatter로 **상태·scope·의존성** 등을 기록하고, 다음 단계에서 큰 작업 여부를 판단할 수 있게 넘긴다.

#### 3. 큰 작업인가?

시스템이 **scope**와 복잡도를 본다. 범위가 넓거나 여러 파일에 걸치면 **다중 Task**로 쪼개 돌려주고, 단순하면 **단일 Task**로 그대로 돌려준다. 서로 겹치지 않고 **병렬로 돌릴 수 있는 단위**는 병렬 태스크로 묶어 반환한다.

#### 4. Task 승인

분할·검토 결과로 나온 **scope와 태스크 구성**을 사람이 검토한다. 의도와 맞는지, 범위·작업 나눔이 적절한지 보고 **승인하거나 거절**한다.

#### 5. Task 등록 - Pending

승인된 태스크는 **`Pending`** 으로 큐에 올라간다. `orchestrate.sh`가 **의존 관계**와 **scope 충돌**을 확인한 뒤, 안전한 순서로 실행한다.

### 5-2. Task Queue 아키텍처

#### 메인 루프 - orchestrate.sh

`orchestrate.sh`가 무한 루프를 돌며 태스크를 스케줄링한다. 한 사이클은 다음 순서로 진행된다.

```mermaid
flowchart LR
    A[태스크 스캔] --> B[의존 해석]
    B --> C{scope 충돌?}
    C -->|없음| D{슬롯 여유?}
    C -->|있음| E[대기]
    D -->|있음| F[워커 디스패치]
    D -->|없음| E
    F --> G[시그널 대기]
    G --> A
```

1. **태스크 스캔** - `.orchestration/tasks/`에서 pending 태스크를 수집하고 priority(high/medium/low) → depends_on 순으로 정렬한다.
2. **의존 해석** - `depends_on`에 명시된 선행 태스크가 모두 `done`인 것만 큐에 넣는다.
3. **scope 충돌 검사** - 현재 실행 중인 태스크의 scope와 겹치는지 glob 패턴으로 비교한다. 겹치면 순차 실행으로 밀린다.
4. **슬롯 관리** - 최대 N개(기본 2) 슬롯에서 동시 실행. 메모리 압박 시 자동 백오프한다.
5. **시그널 대기** - `fswatch`로 `.orchestration/signals/` 디렉토리를 감시하여 워커 완료를 즉시 감지한다.

#### 태스크 라이프사이클

```mermaid
flowchart LR
    A[pending] --> B[in_progress]
    B --> C[reviewing]
    C --> D[done]
    B --> E[failed]
    B --> F[stopped]
    C --> G[rejected]
    C -->|retry| B
```

- **failed 시 연쇄 중단** - 실패한 태스크에 의존하는 모든 pending 태스크를 `stopped`로 바꾼다.
- **타임아웃** - 30분간 로그 업데이트가 없으면 워커를 강제 종료하고 failed 처리한다.
- **PID 감시** - 매 루프마다 `kill -0`으로 워커 프로세스 생존을 확인한다.

#### 태스크 실행 - job-task.sh

```mermaid
flowchart LR
    A[worktree 생성] --> B[프롬프트 빌드]
    B --> C[Claude CLI 실행]
    C --> D[scope 검증]
    D --> E[시그널 발행]
```

- **worktree 격리** - `git worktree add`로 태스크 전용 복사본을 만든다. 병렬 태스크끼리 파일 충돌이 원천 차단된다.
- **프롬프트 빌드** - 태스크 내용 + scope 파일(300줄 미만은 전문 임베드, 초과 시 시그니처만) + context 파일 + role 프롬프트를 조합한다.
- **Claude CLI 호출** - `claude --dangerously-skip-permissions --output-format stream-json`으로 실행. 대화 전체가 jsonl로 저장된다.
- **scope 검증** - 실행 후 `git diff`로 scope 밖 변경이 있으면 자동 원복한다.
- **시그널 발행** - 완료 시 `task-done`, 실패 시 `task-failed` 파일을 signals/ 디렉토리에 생성한다.

#### AI 리뷰 - job-review.sh

```mermaid
flowchart LR
    A[리뷰 시작] --> B{승인?}
    B -->|APPROVE| C[머지]
    B -->|REJECT| D{재시도 가능?}
    D -->|예| E[피드백과 재실행]
    D -->|아니오| F[failed]
```

- 작업 완료 후 **별도 Claude 세션**(haiku 모델)으로 코드 변경사항을 검증한다.
- 승인 시 태스크 브랜치를 `main`에 `--no-ff` 머지. 충돌 발생 시 Claude가 자동 해결을 시도한다.
- 거절 시 리뷰 피드백을 포함해 **최대 3회** 재실행한다. 태스크당 누적 비용이 $5를 넘으면 서킷 브레이커가 작동해 즉시 failed 처리한다.



#### 시그널 기반 IPC

각 단계의 완료/실패는 `.orchestration/signals/` 디렉토리에 파일을 생성하여 메인 루프에 알린다. `mkdir` 기반 POSIX 락으로 원자적 생성/소비를 보장한다.

| 시그널 | 의미 |
|--------|------|
| `task-done` | 태스크 실행 성공, 리뷰 단계로 이동 |
| `task-failed` | 실행 오류, 의존 태스크 연쇄 중단 |
| `review-approved` | 리뷰 승인, 자동 머지 진행 |
| `review-rejected` | 리뷰 거절, 피드백 포함 재실행 |

### 5-3. 대시보드 주요 기능

| 페이지 | 기능 |
|--------|------|
| **Overview** | In Progress / Pending / Done / Failed 현황 요약, 활성 태스크 목록 |
| **Tasks** | 전체 태스크 목록, 상태 필터, 의존관계 그래프(DAG) 시각화 |
| **Task Detail** | 태스크 내용, 실행 로그(실시간 WebSocket), 터미널, AI 대화 내역, 비용 |
| **Cost** | 태스크별/모델별 토큰 사용량 및 비용 추적 |
| **Monitor** | 시스템 리소스(CPU, 메모리), 프로세스 상태 |
| **Night Worker** | 야간 워커 설정 및 실행/중지 제어 |
| **Notices** | 태스크 완료/실패/거절 알림 피드 |
| **Settings** | API 키, 모델, 병렬 수, 리뷰 설정 등 |

#### Night Worker — 야간 자동 점검

```mermaid
flowchart LR
    A[Night Worker] --> B[코드 스캔]
    B --> C[이슈 발견]
    C --> D[TASK 자동 생성]
    D --> E[자동 실행]
```

- 지정 시간(기본 07:00)까지 typecheck, lint, 미사용 코드, 테스트 갭 등을 라운드 로빈으로 스캔한다.
- 발견된 이슈를 TASK-*.md로 자동 생성하면 orchestrate.sh가 즉시 픽업한다.
- 예산 한도, 태스크 수 한도로 비용을 제어한다.

### 5-4. 주요 설계 결정

| 결정 | 이유 |
|------|------|
| git worktree로 태스크 격리 | 브랜치 전환 없이 병렬 작업 가능, 충돌 원천 차단 |
| 파일 기반 시그널 IPC | 프로세스 간 의존성 최소화, bash에서 구현 간단, 디버깅 용이 |
| AI 리뷰 필수 | 사람 없이 24시간 운영하려면 자동 검증이 필수 |
| scope 충돌 검사 | 같은 파일을 동시에 수정하는 것을 원천 방지 |
| 모델 자동 선택 | 단순 태스크는 haiku(저비용), 복잡한 태스크는 sonnet으로 비용 최적화 |
| 마크다운 + SQLite 이중 구조 | bash 스크립트 호환성(파일) + 대시보드 성능(DB) 양립 |
| 서킷 브레이커 | 태스크당 $5 한도로 무한 재시도 비용 폭발 방지 |

## 6. 회고

### 6-1. 성과

- **1,400+ 커밋**, **340+ 태스크** 자동 처리 (2026-03 ~ 2026-04)
- 야간에 Night Worker가 코드 스캔 → 태스크 생성 → 실행 → 리뷰 → 머지까지 무인 운영
- 태스크당 평균 비용 $0.04~$0.18 (haiku 기준), 복잡한 태스크도 $0.88 이내

### 6-2. 장점

- **24시간 무인 운영**: 밤새 Night Worker가 typecheck, lint, 코드 품질 이슈를 자동 발견하고 수정. 아침에 출근하면 PR 없이 main에 머지 완료.
- **인간 병목 제거**: AI 리뷰 + 자동 머지로 사람이 코드 리뷰할 필요 없음. 사람은 방향 설정과 최종 확인만.
- **비용 투명성**: 태스크별 토큰 사용량, 비용이 DB에 기록되어 대시보드에서 실시간 확인. 예산 초과 시 자동 중단.
- **장애 자동 복구**: PID liveness 체크, in_progress 타임아웃, zombie 태스크 감지 등 운영 중 발생하는 다양한 장애 상황에 대한 자동 복구 로직 탑재.

### 6-3. 단점

- **bash 기반의 한계**: 서브프로세스 과다 생성, 8진수 시간 버그 등 bash 특유의 문제. Node.js 엔진 전환이 필요.
- **ID 충돌 문제**: Night Worker 다중 인스턴스 실행 시 같은 TASK ID로 파일이 생성되는 경쟁 조건 발생.
- **orchestrate.sh 재시작 시 상태 유실**: 실행 중이던 태스크의 시그널이 소비된 후 프로세스가 죽으면, 재시작 시 해당 태스크가 zombie 상태로 남음.
