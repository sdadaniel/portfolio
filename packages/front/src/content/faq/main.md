# FAQ

자주 묻는 질문과 답변

## 강점과 약점에 대해 설명해주세요

**하드 스킬**로는 하이브리드 앱(App·WebView), React Native, 웹 프론트엔드, 운영 자동화 등 다양한 스택과 환경을 경험해 왔고, 문제의 성격에 맞춰 필요한 기술을 빠르게 파악하고 적용하는 편입니다. Figma 기반 협업에서는 단순히 시안을 전달받아 구현하는 수준을 넘어, 컴포넌트와 디자인 시스템 관점에서 제약사항과 대안, 일관성을 근거로 피드백하며 디자이너와의 커뮤니케이션 비용을 줄여 왔습니다.

**소프트 스킬**로는 요구사항과 제약, 우선순위를 문서와 합의로 정리해 관계자 간 오해를 줄이고, 마찰이 있는 환경에서도 목표 중심으로 협업을 이끌어갑니다. 개발, 기획, 마케팅 사이처럼 경계가 애매한 영역을 '내 일이 아니다'라고 두지 않고 필요한 만큼 학습해서 연결하는 것을 선호하고, 팀 효율 관점에서는 병목이 되는 작업을 먼저 파악하고 정리해 동료가 불필요하게 야근하지 않도록 반복적·수동적인 작업이 쌓이지 않게 업무 흐름을 개선하려고 합니다.

**약점**으로는 초기에 디테일 QA에서 실수가 나는 편이었고, 2~3번 확인해도 놓치는 케이스가 있었습니다. 이를 줄이기 위해 작업 완료 전 QA 체크리스트로 직접 검증하고, 핵심 플로우는 재현 절차와 스크린샷 등 증적을 함께 남기는 식으로 보완하고 있습니다.

## 가지고 있는 경험을 알려주세요

**힘들었던 프로젝트**는 플라잉캣에서 백엔드(Node.js) 개발자로 근무하던 중 프론트엔드 인력 공백으로 React Native 앱 개발에 전환 투입됐을 때입니다. 당시 웹 React 경험은 없었고(Vue만 있었음), 짧은 시간 안에 React와 React Native를 학습하면서 실제 서비스 개발과 운영 품질을 맞춰야 했습니다.

투입 직전 약 2주 동안 핵심 컴포넌트, 상태 관리, 네비게이션, 빌드 및 배포 흐름을 집중적으로 학습한 뒤 곧바로 개발에 합류했습니다. 운영 과정 이슈를 빠르게 흡수·수정하며 안정화에 집중한 결과, 단기간에 운영과 배포까지 단독으로 책임질 수 있는 수준으로 성장했고, 이후 새 스택·도메인 진입 장벽도 많이 낮아졌습니다.

**개선 경험**으로는 번들 용량이 커지며 배포 산출물이 불어나고, 설치(yarn install)·실행·로딩까지 전반이 느려졌던 처리입니다. 번들 분석 도구(bundle visualizer)로 용량 기여도가 큰 모듈과 스타일을 확인해 병목을 특정했습니다.

CJS에서 ESM으로 전환하고 트리 셰이킹을 적용했으며, 잘못된 Sass 사용으로 과다 포함되던 CSS를 정리했습니다. dynamic import와 code splitting으로 초기 로딩에 필요한 코드만 내려받게 구성했고, 그 결과 번들과 자산 용량을 30% 이상 줄이며 설치·로딩 체감을 크게 개선했습니다. 이후에도 성능 이슈를 데이터 기반으로 진단·반복 개선할 기준을 갖게 되었습니다.

## 어떤 회사를 선호하시나요?

여러 차례 경영 악화로 조직이 흔들리는 상황을 겪어 **재무·사업 기반이 안정적인 회사**를 우선합니다. **야근**은 업무 상 필요하면 감당할 수 있습니다. **일에 진지하고 몰입하는 동료**가 있는 팀에서 함께하고 싶고, 팀 안에 **나를 믿고 응원해 주는 동료가 한 명이라도 있으면** 그만큼 더 힘을 낼 수 있다고 느낍니다.

**스택**으로는 Vue보다 React, React Native보다 웹 React 쪽을 선호합니다. **디자인 시스템**이나 컴포넌트 일관성을 다루는 일, **디자이너와 맞춰 가며 논의하는 협업**을 좋아합니다. 어려운 기술 과제를 파고드는 편이라 **단순 반복 위주보다** 새로 배우고 시도할 **도전 여지가 있는 환경**을 선호합니다. 

반대로 개발 자율성이 낮고, 회사가 정한 스택만 허용되는 식으로 도구·기술 선택을 총체적으로 묶어 두는 조직은 지양하는 편입니다.


## 향후 어떤 개발자가 되고 싶은가요?

AI가 빠르게 보급되면서 개발 시장과 업무 방식도 크게 바뀌고 있다고 느낍니다. 프론트엔드 구현만으로는 한계가 올 수 있다고 보아, 최근에는 **AI 관련 학습을 업무와 병행**하며 기술을 넓혀 가고 있습니다. 당분간은 **더 많은 것을 배워야 하는 시기**라고 생각합니다.

**Transformer**, **RNN** 같은 LLM의 기초가 되는 개념을 학습하고 있고, **Python**으로 **챗봇**을 만들며 실습했습니다. 현재는 **Agent 오케스트레이션(orchestrate)** 프로젝트로 **워크플로 자동화·개선**에 집중하고 있습니다.

앞으로도 **AI를 활용하는 역량**을 강화하고 싶습니다. 도구 사용이 당연해져 상향 평준화 환경이 되면, 순수 코딩 못지않게 **AI로 무엇을 만들고 어떻게 운영할지**를 설계·주도하는 개발자로 성장하고 싶습니다.

## 지금까지 다닌 회사의 퇴사 사유가 무엇인가요?

**펀블**에서는 회사 경영이 악화되며 조직 전체를 대상으로 권고사직이 진행되어 그에 따라 퇴사하게 되었습니다.

**산타**에서는 경영 환경 변화로 사무실이 신논현에서 판교로 이전했고, 왕복 약 3시간에 달하는 통근 거리와 시간을 현실적으로 감당하기 어려워 퇴사를 결정했습니다.

**플라잉캣**에서는 경영 악화와 사업 피봇으로 조직 규모가 크게 축소되었습니다. 초기 5명에서 최대 100명 가까이 확장되었다가 다시 5명 이하로 줄어드는 과정을 겪었고, 커리어 방향을 재정비하기 위해 이직했습니다.

**어바웃굿즈**에서는 1인 개발 체계에서 맡은 역할이 고착되며 성장과 협업 환경에 한계를 느껴 퇴사를 결정했습니다.

**아이더마바이오**에서는 개발과 디자인 등 여러 업무를 병행했지만, 개발 전문성을 더 높이고 싶다는 목표가 생겨 이직을 결정했습니다.
